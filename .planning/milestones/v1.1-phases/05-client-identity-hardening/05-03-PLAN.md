<objective>
Wire the Plan 01 crypto and Plan 02 storage into the running game and the build, closing the end-to-end loop. At startup, load/migrate the identity (D-04); after a run, if the identity is valid, compute the submission HMAC with the build-baked secret and POST it in the `"signature"` field (D-07); if the identity is tampered/invalid, skip submission and surface a small "Score not saved" line on the game-over screen (D-05/D-06). The build bakes the gitignored shared secret into the bundle non-literally (D-09/D-10), and the repo ships only a placeholder.

Purpose: This is the integration wave — it turns the tested primitives into observable end-to-end behavior (success criteria 1, 3, 4 become live in the app) and makes the shipped exe carry the secret safely. It is the last wave because it consumes both prior plans.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client submission → Phase 4 server | The server is the real enforcement boundary; the client signature is what makes a normally-played score acceptable once REQUIRE_SIGNATURE flips |
| embedded secret → shipped exe | The build bakes the shared secret into a distributable artifact (extractable ceiling accepted) |
| tampered identity → submit path | A blocked submit must still leave the game fully playable (graceful-degrade) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | Spoofing | main.py signed submission → server | mitigate | Sign every valid submission with `sign_submission` over the locked canonical message (D-07) using the build-baked secret; the Plan 01 oracle test already proved the server accepts it. Server-side `verify_signature` + `REQUIRE_SIGNATURE` (operator flip after ship) is the enforcement boundary. |
| T-05-10 | Tampering / Repudiation | submit gate on TAMPERED identity | mitigate | A tampered identity (D-05) skips submission entirely and shows the game-over notice (D-06); the player cannot push an altered identity onto the board, and the game stays playable (graceful-degrade). |
| T-05-11 | Information Disclosure | secret extraction from exe | accept (mitigated to ceiling) | Secret baked non-literally (byte-array/XOR/split — D-10) so `strings`/grep does not surface it; gitignored at build (D-09) so it is never committed. Accepted residual: a determined reverse-engineer extracts it — inherited Phase 4 ceiling; the server signature check is the boundary. Recorded, not a gap. |
| T-05-12 | Information Disclosure | secret leaked into git | mitigate | `hmac_secret.local` gitignored; repo ships only `hmac_secret.example` placeholder + a `.gitignore` entry; build fails loudly if the local secret is missing (no silent placeholder ship). `git check-ignore` is an acceptance gate. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs — wiring is stdlib-only (`urllib`, `json`, `hashlib`/`hmac` via the Plan 01 module). PyInstaller is an existing dev-only build dep already in `.venv` (no new install task introduced; no RESEARCH package-legitimacy table required). |
