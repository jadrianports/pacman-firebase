<objective>
Rework client identity storage to satisfy IDENT-01/02/03: relocate it out of the game folder into `%LOCALAPPDATA%\PacMan\`, consolidate the two legacy plaintext files into a single obfuscated + HMAC-signed blob, migrate existing players seamlessly (migrate → write → verify-read → delete legacy), and detect out-of-band tampering on load with a fail-closed sentinel that blocks submission without ever silently regenerating the identity.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk identity file → game load | Untrusted: a user/prankster with PC access can edit the blob out-of-band |
| legacy plaintext files → new blob (migration) | One-time read of trusted-local legacy data; must not lose identity on a mid-migration crash |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering | identity.dat (out-of-band edit) | mitigate | File-integrity HMAC (Plan 01 verify_identity_blob) + fail-closed TAMPERED sentinel that blocks submit (D-05); never auto-regenerate on a broken HMAC (would let tampering force-reset / silently move a bit-rotted identity to a new slot). |
| T-05-06 | Information Disclosure | identity.dat plaintext readability | mitigate | Consolidated blob obfuscated (XOR+base64) so initials/machine_id are not greppable/hand-editable (IDENT-02, D-10). Accepted residual: reversible by a determined reader (light-obfuscation ceiling). |
| T-05-07 | Denial of Service / data loss | migration write-then-delete | mitigate | D-04 safety ordering: write new blob → verify it reads back equal → ONLY THEN delete legacy files. A mid-migration failure leaves legacy intact, never losing the identity. |
| T-05-08 | Spoofing (self-reset) | missing-file fresh mint | accept | A user deleting their identity file → fresh machine_id → new slot is the irreducible identity limit (Phase 4 accepted ceiling). "Permanent initials" is permanent-per-machine_id, server-enforced; the file-HMAC detects third-party tampering, not self-initiated reset. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs — storage rework is stdlib-only (`json`, `uuid`, `os`, `re`) plus the Plan 01 client crypto module. No legitimacy checkpoint required. |

**Accepted/deferred (not gaps):** Local-file encryption beyond obfuscation+HMAC is explicitly out of scope (REQUIREMENTS § Out of Scope — client secrets are extractable; the server is the boundary). Stronger local key derivation (HKDF / separate local key) rejected as theater (CONTEXT Deferred Ideas).
</threat_model>
