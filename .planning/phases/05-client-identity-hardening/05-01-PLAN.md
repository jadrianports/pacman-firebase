<objective>
Create the client-side crypto module — the foundation every other Phase 5 file consumes. It mirrors the Phase 4 server's wire-format contract byte-for-byte so client-signed submissions pass the live server's verifier, and it adds the two new primitives this phase needs: a light-but-real reversible obfuscation scheme (IDENT-02) and a domain-separated file-integrity HMAC (IDENT-03, D-08).

This plan closes the single most important loop of the phase: success criterion 4. The test proves — without a live deploy — that a client signature over a known submission passes the real server `verify_signature` (used as a test oracle).

Purpose: Establish the crypto contract before any consumer is built (interface-first ordering — Plan 02 signs the identity blob with this module, Plan 03 signs submissions with it). Concentrating all crypto in one tested module keeps `canonical_message` byte-for-byte aligned with the server and gives the obfuscation/domain-separation logic a single home.
Output: New `leaderboard_crypto.py` at repo root (client copy) + `tests/test_client_crypto.py`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client crypto → server verifier | The HMAC the client produces must be recomputable by the server over identical canonical bytes; any divergence silently breaks accepted submissions |
| obfuscated blob → on-disk file | Obfuscation only resists casual reading; it is not a security control (the HMAC is) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | sign_submission vs server verify_signature | mitigate | Reproduce canonical_message byte-for-byte (D-07); prove with the oracle test that the client sig passes the real server verify_signature with a shared secret (success criterion 4). Server is the real enforcement boundary. |
| T-05-02 | Tampering | sign_identity_blob / verify_identity_blob | mitigate | HMAC-SHA256 over a domain-prefixed message, constant-time compare, fail-closed on non-string/non-ASCII (mirrors server CR-01 totality). Consumed by Plan 02's tamper sentinel. |
| T-05-03 | Elevation/Reuse (cross-protocol) | file sig vs submission sig | mitigate | Domain separation via `IDENTITY_FILE_PREFIX` (D-08) — file and submission messages occupy disjoint message spaces, so a captured file sig cannot be replayed as a submission sig or vice-versa. Tested. |
| T-05-04 | Information Disclosure | obfuscate/de_obfuscate | accept | Obfuscation (XOR+base64) only defeats `strings`/grep/hand-editing (D-10 light-obfuscation altitude). Accepted residual: a determined reverse-engineer reverses it — inherited Phase 4 ceiling; the server-side signature check is the boundary. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan — client crypto is stdlib-only (`hashlib`, `hmac`, `json`, `base64`). No legitimacy checkpoint required. |

**Accepted/deferred (not gaps):** Replay protection is explicitly out of scope (Phase 4 D-04 — "only-if-higher" submit makes replay near-harmless; no nonce/timestamp). Secret extraction from the shipped exe is mitigated only to the casual ceiling here (the embedded-secret non-literal storage lands in Plan 03); the irreducible residual (a decompiler extracts it) is the accepted Phase 4 ceiling, with the server `REQUIRE_SIGNATURE` gate as the boundary.
</threat_model>
