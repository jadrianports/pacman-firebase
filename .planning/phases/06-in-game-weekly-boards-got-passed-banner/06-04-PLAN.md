## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| game client → get_leaderboard (HTTP) at launch | Short-timeout best-effort read; the response feeds only a cosmetic banner. |
| public board response → got-passed banner | Untrusted initials/scores from the board are displayed in the banner; no trust is placed in them beyond display. |
| local marker file → launch compute | Unsigned local state (Plan 03) read to compute passers; tampering is harmless. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-12 | Spoofing | spoofed got-passed rival data (a forged/MITM board response naming fake passers) | accept | A wrong banner is HARMLESS (Phase 5 D-03, RESEARCH "Specifics") — it can only over- or under-name passers; it cannot alter the player's score, submission, or any server state. The player tracks their own best LOCALLY (D-11) and never trusts the board to identify themselves (machine_id stripped, D-10/D-19). Building identity matching to harden a cosmetic line would be theater. |
| T-06-13 | Denial of service | a hung/dead network blocking startup before the menu | mitigate | The launch fetch uses `BANNER_FETCH_TIMEOUT_SECONDS` (~2s, D-09), never the 10s default; `get_leaderboard` swallows the timeout to None → banner silently skipped. Single blocking call BEFORE the menu loop, no threading, no retry — a dead network adds at most ~2s then proceeds (Pitfall 4). |
| T-06-14 | Information disclosure | banner / board leaking other players' private data | mitigate | The board projection is `{initials,score}` only (Phase 4 D-10, relied on unchanged) — the banner shows initials already public on the board. No machine_id, no PII. |
| T-06-15 | Tampering | marker baseline manipulation to fake/suppress the banner | accept | Same as T-06-08 (Plan 03) — the marker is unsigned by design (D-13); editing it only changes a cosmetic banner. No security impact. |
| T-06-16 | Denial of service | marker write crashing on submit or board-open | mitigate | `write_marker` is best-effort (Plan 03, try/except → silent); a write failure cannot break the submit flow or the board exit. The launch read is likewise guarded. |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs (RESEARCH "Package Legitimacy Audit": N/A — stdlib + already-imported pygame only; no new client dependency, preserving the PyInstaller + stdlib-only contract). |

Existing controls relied upon (do NOT re-litigate): the server is the enforcement boundary (Phase 4 — HMAC verify, sanity ceiling, permanent initials); the client never trusts the board for anything beyond display; your-best is tracked locally not located on the board (D-11/D-19); {initials,score}-only projection (D-10). The CI golden net (ghost-AI traces + determinism guard) is a merge gate — this phase touches no sim/ghost code, so it stays green untouched (do NOT re-bless).
