|----------|-------------|
| Contributor push/PR → main | The required CI check is the gate preventing an unverified (bricked) change from reaching main |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-T | Tampering | a behavior-bricking change reaches main without the net catching it | mitigate | CI runs the full golden/micro/determinism/cloud-fn suite on every push + PR; branch protection makes the green 'test' check required to merge to main (D-09) |
| T-07-E | Elevation of Privilege | CI workflow grants broad token permissions or runs untrusted code with secrets | accept | The workflow needs no secrets (cloud-fn tests are mocked; no deploy step); default read-only token is sufficient; no Firebase creds are exposed to the runner |
| T-07-SC | Tampering | CI installs unpinned/poisoned deps | mitigate | requirements-dev.txt is pinned (D-20) and was human-verified in Plan 01; setup-python and pygame are pinned to the D-08 bless versions; action majors are pinned |

No new runtime attack surface ships in the .exe — the workflow is a dev/CI artifact only; it builds and
tests, it does not deploy or alter the shipped game.
</threat_model>
